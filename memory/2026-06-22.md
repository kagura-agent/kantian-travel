## Study Apply (10:16)

**Mode**: Apply | **Source**: quality-proof-for-optimizations gradient (2026-06-21) + tokdiet shadow-eval pattern

**Applied:** Signal preservation quality check to `tools/compress-output.sh`:
- `signal_preservation_check()` extracts critical signals (errors, warnings, test failures, aggregate summaries) from raw content
- Verifies each signal survives compression via grep -qF
- Logs `signal_preserved_pct`, `signals`, `signals_kept` to compress-metrics.jsonl
- Emits stderr warning when preservation < 80%

**Immediate payoff (bugs found on first run):**
1. Test compression regex was case-sensitive: `ERROR|error` missed `Error:` — mixed-case error messages silently dropped for months
2. Build compression used PCRE lookahead `(?!...)` in ERE mode — grep warnings on every build run

**Key insight:** Shadow-eval is more valuable as a one-time bug-finder than as ongoing monitoring. First measurement pass always reveals silent degradation. Once fixed, monitoring becomes a guard rail. Implication: add quality checks to ALL our transformation tools (compress-daily-memory.sh, wiki compression, search reranking) as one-time audit passes.

**Pattern:** "Measure-then-trust" — any lossy transformation claiming quality-neutral MUST prove it. The first run of measurement invariably finds bugs. Tool-makers are blind to their own tool's failures because they never instrument the output quality.
