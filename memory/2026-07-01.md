**Problem**: Issue selection in workloop checks GitHub for closed PRs (Gate 3 in issue-funnel.sh) but never checks wiki-level knowledge (pr-superseded-lessons.md) which documents WHY approaches failed. Even when wiki has 17+ documented failure patterns for a repo, the agent blindly selects issues without awareness.

**Applied**:
1. Created `tools/prior-failure-check.sh` — standalone script that checks: (a) pr-superseded-lessons.md for repo-level documented patterns, (b) wiki/projects/ notes for failure mentions, (c) GitHub closed PRs for issue-specific and repo-level history. Outputs warnings with specific patterns to avoid. Supports single repo, repo#issue, and --all modes.
2. Integrated wiki failure pattern check into `issue-funnel.sh` as new quality signal between "wiki notes bonus" and "low open-PR count bonus". Repos with >5 documented failures get -20 score penalty; 1-5 get -10 penalty. Warning output during funnel evaluation.

**Verification**:
- NemoClaw: 17 wiki hits → -20 penalty applied ✅
- Ornith (no history): 0 hits → no penalty ✅
- Vercel/ai: 5 hits → -10 penalty ✅
- Regression gate: no regressions ✅

**Behavioral change**: When selecting issues, repos with documented failure patterns are now scored lower AND show explicit warnings. Prior-failure-check.sh can also be called standalone for manual issue selection (bypassing the funnel). This addresses the structural gap: "I have all the knowledge about why I failed, but I don't consult it before trying again."

**Ornith 3-layer mapping**: Layer 1 = DNA red lines (immutable). Layer 2 = this script (deterministic check). Layer 3 = agent judgment informed by the output.
