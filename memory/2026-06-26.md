## Study Apply (09:30)

**Applied: hn-scan.sh Algolia API fix (hn-scan-broken-signal root cause)**

- **Problem**: Algolia HN API silently dropped `points` from `numericAttributesForFiltering`. Our `hn-scan.sh` sent `points>N` in numericFilters → 400 error → empty response → "No stories found" → all study scout/quick-scan HN data was zero for days.
- **Root cause**: Server-side filter no longer supported; script treated API error as "no results."
- **Fix**:
  1. Removed `points` from server-side numericFilters (only `created_at_i` remains)
  2. Added client-side jq filtering: `[.hits[] | select((.points // 0) >= $min)]`
  3. Added explicit API error detection (checks for `.message` field in response → exits with clear error)
- **Behavioral change**: Before: 0 results on every HN scan. After: proper results (tested: 2 matches "ai agent" 3d, 3 matches "coding agent" 7d)
- **Verification**: ran hn-scan.sh with multiple queries, regression-gate passed

- **Why this over other apply targets**: unapplied.md backlog fully cleared (all [x]). This was the highest-signal structural fix available — directly addressing a preflight violation with measurable behavioral change.

### Study Reflect 09:30

**成功分析:**
- goal: Fix a broken tool that's been causing silent study degradation for days
- approach: Traced the actual API call, found 400 error (points no longer in numericFilters), moved filter client-side
- key_decisions: (1) Investigated root cause instead of switching to web_search fallback. (2) Added error detection for future API changes.
- pattern: When a tool returns "no results" consistently, the first debugging step should be checking the HTTP response, not the query parameters. Silent API contract changes need explicit error detection.
- applies_when: Any tool wrapping a third-party API that starts returning empty — always check if the API itself changed, don't assume the query is wrong.

**失败分析:**
- failure_point: This issue existed for multiple days before being fixed. The hn-scan-broken-signal gradient was logged on 06-25 but the advice was "switch to alternative source" rather than "fix the tool."
- flawed_reasoning: Treating the symptom (no results → use alternative) instead of the disease (API contract change).
- what_should_have_been_done: First time HN scan returns 0 results in a scout, add `-v` to curl and check the actual HTTP response.
- prevention_principle: Gate scripts and API wrappers should capture and report HTTP status codes / error messages, not just check if output is empty.

**Gradient 自检:**
- Tool friction: Yes — the 400 error was invisible because curl's stderr was discarded (`2>/dev/null`). The fix adds explicit error detection but a broader principle applies: never silence stderr for diagnostic tools.
- No gradient needed — the fix itself IS the structural change. The broader pattern (API error detection) is already encoded in the script.
