### Deep Read: SmallCode Hybrid Search Implementation

**Architecture**: `hybrid_search.js` (254 lines) — lightweight, zero-dependency approach:
- **Tokenizer**: Regex-based, splits camelCase/snake_case identifiers, filters stopwords
- **"Embedding"**: FNV-1a hash of tokens into sparse bag-of-words vector (not neural — hashed term frequency), L2-normalized
- **Cosine similarity**: Sparse vector dot product (iterates smaller keyset)
- **BM25**: Standard k1/b parameterized BM25 over term frequency maps
- **Fusion**: `score = bm25 + vectorWeight * cosine + exactBoost * (exact ? 1 : 0)`
- **Chunking**: Regex-based symbol detection (function/class/method boundaries), 80-line cap per chunk, no real AST parser
- **Scale**: 1500 files max, 512KB max per file, built fresh per search (no persistent index)

**Key insight**: This is NOT true semantic search — it's enhanced BM25 with hash-collision-based soft matching. The "semantic" component only catches term overlap after identifier splitting (e.g., `getUser` ↔ `fetchUserData` via shared `user` token). It won't find `authenticate` when searching `login` unless they share subwords. But for code search this is often sufficient because code identifiers ARE the semantics.

**Tradeoff**: Zero dependencies + instant startup vs. quality ceiling. Good enough for small codebases (the target), but scales poorly (rebuilds full index per search, O(files × chunks) on every call). Appropriate for SmallCode's constraint: "fully local with zero external services."

**Issue #76 insight**: Real users hitting llama.cpp 300s timeouts on larger projects. Terminal freezes after timeout — the TUI doesn't gracefully recover from model timeout during multi-tool flows. Reveals the fundamental tension: small local models are slow on non-trivial tasks, and the tooling assumes fast responses.

**Relevance to us**: LOW for direct adoption (we have access to frontier models), but the hashed-BoW-as-pseudo-embedding pattern is clever for resource-constrained contexts. The symbol-boundary chunking without a full parser is a good 80/20 approach worth remembering.
