# 看天出发 — 产品文档

## 一句话

好天气不该浪费。

## Slogan

**「此刻合适，也适合你」**

## 核心洞察

传统旅游攻略是**预制的、静态的**——写攻略的人不知道你去的那天天气怎样、季节对不对、花开了没有、海鸥来了没有。用户拿着攻略出门，发现天气不对什么都看不到。

我们的方案是**实时生成的**——先看天气和当前条件，再做方案。用户打开 app 看到的每一个方案，都是"此刻去就对了"。

### 用户痛点（小红书真实提问）

用户在手动做我们要自动化的事：

- "我下周去海南，有晴天吗？" → 天气
- "现在阿勒泰的草黄了吗" → 物候
- "养马岛天晴么" → 天气 + 值不值得去

本质都是同一个问题：**目的地 + 实况 + 时间 → 现在去合不合适？**

我们把这个问答关系**反过来**：用户不需要问，打开就看到答案。

## 核心公式

```
目的地知识 × 目的地实况 × 你的需求 = 此刻适合你的旅行方案
```

不满足条件的方案**不存在**——直接不出现。用户看到的就是答案。

### 目的地实况

- **天气**：晴雨、温度、风力、能见度、日出日落
- **物候**：花期、红叶、候鸟、萤火虫、果季
- **自然条件**：潮汐、浪高、水位、积雪、云海概率
- **交通**：拥堵、道路封闭、轮渡和公共交通状态
- **人流**：景区客流、预约余量、排队时间
- **营业状态**：临时闭园、维护、演出场次、活动变更
- **本地事件**：市集、节庆、展览和临时活动
- **价格与供给**：门票、酒店价格、库存和预约情况

### 你的需求

由两部分组成：**位置 + 约束**。

**位置**：用户所在区级定位（苏州-姑苏区、苏州-吴江区等）。同一个城市不同区到目的地的车程可能差30分钟以上，影响“值不值得去”的判断。交通时间通过路线规划 API（高德/百度）实时计算，不手动维护——同时可拿到实时路况。精确导航交给高德，我们只提供“约1.5h车程”这样的决策参考。

**约束**（通过 tag 或自由输入）：
- 「周末带狗1天」= 时间(周末) + 同伴(宠物) + 时长(1天)
- 「下雨天室内遛娃」= 天气(雨) + 同伴(小孩) + 场景(室内)
- 「2人浪漫不开车」= 人数(2) + 氛围(浪漫) + 交通(公共)
- 「下周带爸妈两日游，走路不能太多」= 时间 + 同伴 + 体力约束

约束影响生成：
- **POI 筛选**：带狗 → 只选宠物友好的地方
- **路线规划**：不开车 → 只选高铁/公交可达
- **时间安排**：带小孩 → 节奏慢、不能太早出发
- **活动选择**：浪漫 → 偏向咖啡/日落/小众，不选游乐园

前端预设标签是**快捷入口**，帮懒人省去打字。用户也可以自由输入约束。

### 时间标签的两种生成模式

**模式 A：定日期生成**
> 「现在」「明天」「这周末」「下周末」
> → 拿确定日期的实况 → 生成
> 输出：`"validFor": "2026-07-25"`

**模式 B：找窗口生成**
> 「请3天假」「小长假5天」「找7天连晴」
> → 扫描未来15天预报 → 找连续N天适合的窗口 → 生成
> 输出：`"validFor": ["2026-07-28", "2026-08-03"]`

## 架构

### 分发模型

**核心洞察：同城市 + 同 tag 的用户看到一样的方案。**

生成按"城市 × tag"组合跑，不按用户数：
- 100 个苏州用户选"明天"，只需要生成一次
- 预设 tag 全是提前生成的静态 JSON
- 只有自定义约束（自由输入）才需要实时调模型

这意味着**生成成本可控**——按城市×tag 组合生成，不按用户数。预设 tag 提前生成，结果可 CDN 缓存；自定义约束才需实时调模型。

后端负责：
- 定时拉取实时数据（天气、物候、人流 API）
- 跑生成流程（规则筛选 + 调模型）
- 存储和分发生成结果
- 用户自定义约束的实时生成
- 用户数据（收藏、历史、偏好）

```
generated/
└── 苏州/
    ├── 明天.json           ← 包含多个方案的数组
    ├── 这周末.json
    ├── 周末带狗1天.json
    └── 找7天连晴.json
└── 上海/
    ├── 明天.json
    └── ...
```

用户打开 app → 读取位置（苏州）→ 点 tag（明天）→ 拉对应 JSON → 展示。

### 架构：动态发现 + 两次模型

**核心转变：没有静态知识库。** 目的地不是提前维护的列表，而是根据用户位置+需求动态搜索发现的。

**生成流程（两次调模型）：**

```
第一次调用（轻量，决定搜什么）：
  输入：用户位置 + tag + 约束 + 当天天气概况
  输出：搜索策略 JSON
  {
    "poi_types": ["公园", "湖泊", "徒步路线"],
    "transport": "driving",
    "radius_km": 100,
    "exclude": ["室内", "需要预约"],
    "time_preference": "早出晚归"
  }
  → 便宜（输入短、输出小 JSON）
  → 同样约束的策略可缓存复用

  执行搜索：
  按策略调高德 API → 发现景点 + 计算路线 + 搜餐饮
  拉天气（实时）
  查质量层（用户反馈、积累数据）

第二次调用（重量，生成方案）：
  输入：搜索结果 + 天气 + 路线 + 质量数据 + 用户约束
  输出：完整方案数组（3-5 个）
```

**为什么两次：**
- 用户需求是自然语言（"带狗周末不想太累"），硬编码规则映射不完
- 搜索策略由模型推导，任何奇怪的 tag 都能处理
- 第一次输出可缓存（同样约束的搜索策略不会天天变）
- 第二次才是重计算，但输入已经被第一次精确筛过

### 数据层：缓存 + 积累

没有静态知识库，只有缓存和积累：

| 数据 | 来源 | 保质期 | 失效风险 |
|------|------|--------|---------|
| POI 基础信息 | 高德搜索 → 缓存 | 30天 | 低（偶尔关闭） |
| 距离/车程 | 高德路线 → 缓存 | 7天 | 低（新修路） |
| 照片 | Google Places → 下载 | 90天 | 中（需匹配季节） |
| 天气 | 天气 API | 不缓存 | — |
| 搜索策略 | 模型输出 → 缓存 | 长期 | 极低 |
| 用户评价 | 用户反馈 → 积累 | 永久 | 极低 |

**冷启动无质量数据也能跑**——模型自身训练数据里有旅游常识。质量层随使用逐步积累，让推荐越来越准。

### 数据库表

```sql
-- POI（动态发现，缓存）
pois (id, amap_id, name, type, lat, lng, discovered_at, last_verified_at)

-- 路线缓存
route_cache (from_location, to_poi_id, distance_km, duration_min, cached_at)

-- 照片
poi_photos (id, poi_id, url, source, season_tag, weather_tag, cached_at)

-- 质量积累
poi_quality (poi_id, condition_type, value, source, created_at)
poi_ratings (poi_id, user_id, score, weather_when_visited, comment, created_at)

-- 搜索策略缓存
search_strategies (constraint_hash, strategy_json, created_at)

-- 生成结果
plans (id, location, tag, valid_date, generated_at, title, reason, weather_json)
plan_days (id, plan_id, day_index, activity, weather_json)
plan_steps (id, day_id, step_index, type, text, start_time, end_time, description, poi_id)
```

### 交通方式影响搜索范围

搜索策略中的 `transport` 决定搜索方式：
- **自驾** → 高德驾车路线，半径 ~150km
- **高铁** → 搜索车站可达城市，半径 ~500km
- **公交** → 高德公交路线，半径 ~30km
- **步行/骑行** → 近距离，5-15km

用户约束自动限定交通方式：
- 「带狗」→ 只搜自驾（高铁不让带宠物）
- 「不开车」→ 只搜公交/高铁
- 「老人」→ 排除大量步行的方案

## 技术选型

### 数据库：PostgreSQL + PostGIS

**为什么：**
- 关系型核心 → plans/days/steps/pois 之间有明确关系
- JSONB → 存 weather、strategy 这种不固定结构，可建 GIN 索引
- PostGIS → 算距离、"找半径100km内的目的地"
- 数组类型 → `tags text[]`，原生支持 `WHERE 'pet-friendly' = ANY(tags)`
- 读多写少，成熟稳定，免费

### API 依赖

| API | 用途 | 部署位置 |
|-----|------|---------|
| 高德 Web 服务 | POI 搜索 + 路线规划 + 天气 | 本地/服务器直连 |
| Google Places Photos | 真实地点照片 | VM1（日本）代理 |
| 大模型 | 搜索策略 + 方案生成 | 按需调用 |

## 设计原则

1. **懒人优先** — 选择越少越好，打开就是答案
2. **实时即信任** — 用户知道方案是基于当前条件生成的
3. **没有废方案** — 不满足条件的不出现
4. **保质期** — 方案会过期，强化"此刻"感
5. **动态发现** — 不维护静态列表，搜索即发现
6. **越用越好** — 质量层随用户反馈积累
