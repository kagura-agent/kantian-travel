# 看天出发 — 产品文档

## 一句话

好天气不该浪费。

## 核心洞察

传统旅游攻略是**预制的、静态的**——写攻略的人不知道你去的那天天气怎样、季节对不对、花开了没有、海鸥来了没有。用户拿着攻略出门，发现天气不对什么都看不到。

我们的方案是**实时生成的**——先看天气和当前条件，再做方案。用户打开 app 看到的每一个方案，都是"此刻去就对了"。

## 核心逻辑

```
目的地知识 × 目的地实况 × 你的需求 = 此刻适合你的旅行方案
```

不满足条件的方案**不存在**，不是标灰、不是提示"天气不佳"——直接不出现。用户看到的就是答案。

### "当地实时信息"包括：

- **天气**：晴雨、温度、风力、能见度、日出日落
- **物候**：花期、红叶、候鸟、萤火虫、果季
- **自然条件**：潮汐、浪高、水位、积雪、云海概率
- **交通**：拥堵、道路封闭、轮渡和公共交通状态
- **人流**：景区客流、预约余量、排队时间
- **营业状态**：临时闭园、维护、演出场次、活动变更
- **本地事件**：市集、节庆、展览和临时活动
- **价格与供给**：门票、酒店价格、库存和预约情况

### "用户约束"包括：

用户约束是生成时的 prompt 约束，不是前端筛选条件。

示例：
- 「周末带狗1天」= 时间(周末) + 同伴(宠物) + 时长(1天)
- 「下雨天室内遣娃」= 天气(雨) + 同伴(小孩) + 场景(室内)
- 「2人浪漫不开车」= 人数(2) + 氛围(浪漫) + 交通(公共)
- 「下周带爸妈两日游，走路不能太多」= 时间 + 同伴 + 体力约束

约束影响生成：
- **POI 筛选**：带狗 → 只选宠物友好的地方
- **路线规划**：不开车 → 只选高铁/公交可达
- **时间安排**：带小孩 → 节奏慢、不能太早出发
- **活动选择**：浪漫 → 偏向咖啡/日落/小众，不选游乐园

前端预设标签是**快捷入口**，帮懒人省去打字。用户也可以自己输入约束，生成系统去理解和匹配。

### 时间标签的两种生成模式：

**模式 A：定日期生成**
> 「现在」「明天」「这周末」「下周末」
> → 拿确定日期的天气 → 匹配 → 生成
> 输出：`"validFor": "2026-07-25"`

**模式 B：找窗口生成**
> 「请3天假」「小长假5天」「找7天连晴」
> → 扫描未来15天预报 → 找连续N天适合的窗口 → 生成
> 输出：`"validFor": ["2026-07-28", "2026-08-03"]`
> 玩法模板可加 `"minDays": 7` 指定最少连续天数

## 架构

### 三层数据结构

**① 目的地知识（完全静态，人工维护）**

地理事实，不因天气变化。

```json
{
  "id": "moganshan",
  "name": "莫干山",
  "region": "湖州",
  "from": { "苏州": { "duration": "2h", "mode": "自驾", "distance": "160km" } },
  "pois": [
    { "name": "裸心谷", "type": "stay", "lat": 30.63, "lng": 119.87, "priceRange": "800-1500" },
    { "name": "竹海栈道", "type": "play", "lat": 30.64, "lng": 119.88 },
    { "name": "山顶观景台", "type": "play", "lat": 30.65, "lng": 119.86 }
  ]
}
```

**② 玩法模板（半静态，关联条件）**

"如果条件满足，可以这样玩"——一个目的地可以有多个玩法模板。

```json
{
  "destinationId": "moganshan",
  "activity": "云海日出",
  "description": "凌晨5点山顶观景台看云海翻涌",
  "conditions": {
    "weather": ["雨后初晴"],
    "visibility": { "min": 10 },
    "season": ["3-11月"],
    "time": "dawn"
  },
  "steps_template": [
    { "type": "depart", "time": "04:30" },
    { "type": "play", "poi": "山顶观景台", "duration": "2h" },
    { "type": "eat", "text": "山间早餐" }
  ]
}
```

**③ 实时数据源（完全动态，API 拉取）**

```json
{
  "date": "2026-07-25",
  "weather": { "苏州": {}, "莫干山": {} },
  "phenology": { "moganshan": ["竹笋季结束", "萤火虫活跃"] },
  "events": { "moganshan": ["音乐节 7/25-7/27"] },
  "traffic": { "苏州→莫干山": { "status": "畅通", "eta": "1h50m" } }
}
```

### 实际生成流程（简化版）

```
1. 用户选 tag（周末带狗1天、找7天连晴、明天…）
2. 系统拿：静态知识 × 实时数据 × 这个 tag
3. 生成属于他的方案列表
4. 前端直接展示，不做任何筛选
```

每天晚上跑一次生成。用户第二天打开看到的就是“今天的答案”。

简单、干净、懒人闭眼用。

```
每晚定时运行：

  Step 1: 规则筛选（快、便宜，省 token）
  ─────────────────────────────────────
  遍历玩法模板 → 用 conditions 做硬过滤
  - 暴雨天 → 干掉所有户外
  - 冬天 → 干掉漂流
  - 非花期 → 干掉赏花
  输出：候选玩法列表

  Step 2: 大模型生成（灵活、有灵魂）
  ─────────────────────────────────────
  输入：候选玩法 + 目的地知识 + 实时数据
  输出：完整方案
  - 组合当天最佳玩法（可能一个目的地拼多个活动）
  - 写出 reason（"明天雨后初晴，云海概率极高"）
  - 调整时间安排（下午雷阵雨 → 户外挪到上午）
  - 生成自然的 tips（"带件薄外套，山顶温差大"）
  - 决定标题怎么写才吸引人
```

**为什么两步：**
- 规则筛选保证不出错（不会暴雨天推露营）
- 大模型保证不生硬（不是机械拼凑，像本地人给建议）
- `conditions` 可以写得粗粒度，模型自己判断边界情况
- conditions 本质是给模型的**提示**，告诉它这个玩法在什么情况下有意义

### 输出层：生成结果（前端展示，每天更新）

```json
{
  "id": "moganshan-20260725",
  "title": "莫干山云海 + 竹林徒步",
  "generatedAt": "2026-07-24T23:00",
  "validFor": "2026-07-25",
  "category": "tomorrow",
  "reason": "明天雨后初晴，云海概率90%",
  "weather": { "icon": "⛅", "temp": "26°", "desc": "雨后转晴" },
  "days": [
    {
      "photo": "...",
      "activity": "...",
      "weather": { "icon": "⛅", "temp": "26°" },
      "steps": []
    }
  ]
}
```

### 关键字段

- `reason` — 告诉用户"为什么是今天"，这是核心卖点
- `validFor` — 方案有保质期，过了就消失
- `id` 带日期 — 同一个目的地不同天生成的是不同方案
- 前端**不做筛选逻辑**，拿到什么显示什么

## 设计原则

1. **懒人优先** — 选择越少越好，打开就是答案
2. **实时即信任** — 用户知道方案是基于当前条件生成的
3. **没有废方案** — 不满足条件的不出现，不标灰不打叉
4. **保质期** — 方案会过期，强化"此刻"感
