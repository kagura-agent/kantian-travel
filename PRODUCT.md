# 看天出发 — 产品文档

## 一句话

好天气不该浪费。

## 核心洞察

传统旅游攻略是**预制的、静态的**——写攻略的人不知道你去的那天天气怎样、季节对不对、花开了没有、海鸥来了没有。用户拿着攻略出门，发现天气不对什么都看不到。

我们的方案是**实时生成的**——先看天气和当前条件，再做方案。用户打开 app 看到的每一个方案，都是"此刻去就对了"。

## 核心逻辑

```
方案 = 好去处 × 实时条件
```

不满足条件的方案**不存在**，不是标灰、不是提示"天气不佳"——直接不出现。用户看到的就是答案。

## Slogan 候选

| 方向 | 候选 |
|------|------|
| 不用想 | 天气替你做决定 / 好天气，不浪费 |
| 实时鲜 | 今天的天气，今天的方案 / 每次打开都是新鲜的 |
| 就现在 | 天气对了就出发 / 此刻最值得去的地方 |
| 现做的 | 刚刚为你做的方案 / 不是攻略，是今天的答案 |
| 过期感 | 明天就不一样了 / 只在今天成立的方案 |
| 过程 | **先看天，再做方案** / 看了天气才做的 |

当前倾向：**「先看天，再做方案」**

## 架构

### 生成流程（后端，每晚运行）

```
天气API + 物候日历 + 目的地知识库 → AI 生成 → 明天的方案列表
```

### 数据结构

**输入层：目的地知识（静态模板）**

```json
{
  "id": "moganshan",
  "name": "莫干山",
  "location": { "lat": 30.63, "lng": 119.87 },
  "activities": [
    {
      "type": "云海日出",
      "conditions": { "weather": "雨后初晴", "visibility": ">10km" }
    },
    {
      "type": "竹林徒步",
      "conditions": { "weather": "非暴雨" }
    },
    {
      "type": "漂流",
      "conditions": { "season": "6-9月", "weather": "晴/多云" }
    }
  ],
  "stays": [],
  "transit": { "from": "苏州", "duration": "2h", "mode": "自驾" }
}
```

**输出层：生成结果（前端展示，每天更新）**

```json
{
  "id": "moganshan-20260725",
  "title": "莫干山云海 + 竹林徒步",
  "generatedAt": "2026-07-24T23:00",
  "validFor": "2026-07-25",
  "category": "tomorrow",
  "reason": "明天雨后初晴，云海概率90%",
  "weather": { "icon": "⛅", "temp": "26°", "desc": "雨后转晴" },
  "days": [
    {
      "photo": "...",
      "activity": "...",
      "weather": { "icon": "⛅", "temp": "26°" },
      "steps": [...]
    }
  ]
}
```

### 关键字段

- `reason` — 为什么推荐这个方案（核心卖点，让用户知道这是实时的）
- `validFor` — 方案有保质期，过了就消失
- `id` 带日期 — 同一个目的地不同天生成不同方案
- 前端**不做筛选逻辑**，拿到什么显示什么

## 设计原则

1. **懒人优先** — 选择越少越好，打开就是答案
2. **实时即信任** — 用户知道方案是基于当前条件生成的
3. **没有废方案** — 不满足条件的不出现，不标灰不打叉
4. **保质期** — 方案会过期，强化"此刻"感
