# 看天出发 (Kantian Travel)

> 看天气，选目的地，出发。懒人旅行决策工具。

## 产品信条

1. **只有选，没有规划** — 用户永远在选，永远不在规划
2. **任何需要用户"想"的步骤都是产品的失败**
3. **打开就有答案** — 零操作冷启动，基于位置直接推荐，不满意就换一批
4. **为懒人服务** — 这是核心，不是附加

## 核心逻辑

```
输入极简：我在哪 + 有几天（默认都不需要输入）
→ 系统生成完整方案（不是地点，是“拿走就能出发”的方案）
→ 用户只做一件事：选
```

**卡片 = 方案，不是地点。**

| 时间窗 | 卡片内容 |
|---|---|
| 2天（周末） | 一个具体目的地的周末方案（怎么去、玩什么、住哪） |
| 3天 | 2-3个点的小路线 |
| 一周 | 完整路线规划，每天在哪干什么都安排好 |

用户不动脑子，拿走就走。

## 默认行为

- 已知用户位置（GPS/上次设置）
- 默认周末2天
- 打开即展示：这周末 + 下周末，附近哪里天气好
- 跟外卖app一样 — 打开就是推荐，不是让你先搜

## 用户场景

### 场景1：周末随机出逃
"周六日没事，想出去走走" — 2天，短途，说走就走

### 场景2：带家人出游
"爸妈来了/放假了，有一周" — 需要舒适，节奏慢，天气更重要（老人怕晒怕冷）

### 场景3：长假卡位
"十一/春节想出去，但还没想好去哪" — 提前看天气趋势，提前订票

### 场景4：天气窗口猎人
"最近哪里有好天气？" — 不限时间，看哪里天气最好，倒推什么时候去

### 场景5：已有目的地，挑日子
"想去黄山，最近哪几天天气好？" — 地点定了选时间

### 场景6：多人凑时间
"我和朋友都只有下周四五有空" — 时间窗口窄，更依赖系统发现

### 共性

用户都懒得研究，但约束组合不同：
- 有时候时间定了选地方
- 有时候地方定了选时间
- 有时候都没定，纯看天气+可达性排列组合

## 内容原则

**推荐的价值 = 信息差。** 用户已经知道的东西不需要推荐。

- 内容要具体、有画面感、给用户新信息
- 不要宽泛的热门景点（西湖、外滩、夫子庙）
- 要小众的、当季特别好的、本地人才知道的
- 哪怕是热门地方，角度也得是新的
- 每条推荐要让人觉得"诶，这个我还真不知道"

举例：
- ✖ "杭州西湖骑行" — 废话
- ✔ "径山寺后山有条野路，竹林里走40分钟到茶田，人极少"
- ✔ "这个季节千岛湖边上的芹川古村有萤火虫"
- ✔ "溧阳一号公路自驾，路边有个不要钱的野温泉"

**内容壁垒 = 内容质量 > 数据覆盖。**

## 筛选轴

1. **出发地** — 我在哪
2. **时间窗口** — 哪几天有空
3. **天气** — 那几天哪里天气好

## 信息层（选完后展开）

- 怎么去（交通方式+时间+票价）
- 玩什么（小红书/攻略/热门内容）
- 住哪（如需过夜）

## 产品形态

**微信小程序** — 怎么简单怎么来

- 移动端优先，碎片时间决策场景
- 打开第一屏是卡片流（不是搜索框、不是地图）
- 每张卡片 = 一个"可以去"的选项：目的地+天气+交通时间+一句话卖点
- 默认按"这周末"排列，往下就是下周末、再下周
- 像刷短视频一样刷目的地

### 时间选择

不用日历，不用模式切换。就是几个常用 tag，每个都是自然语言，一看就懂：

`这周末` `下周末` `请3天假` `找7天连晴`

完整 tag 列表：
`明天出发` `这周末` `下周末` `请3天假` `小长假5天` `找7天连晴`

- “这周末”“下周末” — 明确时间，系统展示这两天的方案
- “请3天假”“找7天连晴” — 系统在未来半月里找天气最好的连续N天窗口
- 每个 tag 本身就是完整的意思，用户不需要理解任何“模式”

### 详情页 = 决策清单（不是攻略）

全是结论，不是信息。看完就能决定去不去。

### 调性

轻、快、不正经。"周末去哪玩的懒人助手"。

文案风格：
- "这周末杭州放晴了，去不去？"
- "南京连晴3天，高铁才1小时"
- "算了这周哪都下雨，宅着吧 ☔"

### MVP 范围

一个页面，打开就能用：
- 自动定位
- 展示这周末+下周末天气好的目的地卡片
- 点进去看交通+玩什么
- 不做账号、不做收藏、不做个性化推荐

### 差异化

市面旅游app都是"你告诉我去哪，我帮你查"。
我们反过来：**你什么都不说，我告诉你能去哪。**

不是搜索引擎，是推荐引擎。核心资产是天气×交通×时间的实时匹配能力。
